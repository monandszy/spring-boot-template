# Test Strategy

## Test Coverage
A metric that represents the degree to which the source code is executed when a particular test suite runs. Rather than chasing an arbitrary 100% line coverage just to appease a tool, coverage should focus on ensuring all critical business rules, edge cases, and module boundaries (Facades) are continuously verified.

### Why?
- **Confidence in Refactoring:** High-quality coverage acts as a safety net. Developers can rewrite internal module logic or upgrade dependencies without fear of secretly breaking existing features.
- **Living Documentation:** Well-written test suites accurately describe what the system actually does right now, which is often more reliable than reading outdated requirements documents.

---

## Facade Tests (Integration Testing)
The primary workhorse of the testing strategy. These tests interact exclusively with the module through its `in` port (the Facade). The core Domain and Services run exactly as they would in production, but external dependencies via the `out` ports (like the Database or LLM APIs) are satisfied using mocks or Fixtures (as defined in the Domain Design phase).

### Why?
- **High ROI (Return on Investment):** Testing at the Facade level covers the entire use case (from input validation to domain rule execution) in a single test, providing the highest ratio of confidence to effort.
- **Implementation Hiding:** Because the test only talks to the Facade, you can completely refactor the internal classes, services, or domain models without breaking the test. 
- **Fast Execution:** By mocking Outbound Adapters (avoiding real database hits), these tests remain incredibly fast, allowing them to be run continuously during development.

---

## Unit Tests (Domain / Service specific logic)
Highly targeted tests focused on a single class or pure function. In the context of the Ports and Adapters architecture, these are reserved for particularly complex pieces of Domain logic (e.g., complex calculations, intricate state machines, or deep validation rules) that would require an unreasonable number of Facade tests to cover every permutation.

### Why?
- **Pinpointing Complexity:** When a specific domain entity has high complexity (many isolated `if/else` branches), unit testing that entity in isolation is much easier than setting up complex Facade test scenarios.
- **Rapid Debugging:** When an isolated unit test fails, the developer immediately knows exactly which function is broken, removing the need to trace through the entire service layer.
- **Pragmatism:** By keeping unit tests strictly "if needed," you avoid the trap of writing brittle, over-mocked tests for simple CRUD services that are already adequately covered by the Facade tests.

---

## Entrypoint Tests (HTTP / REST APIs)
Tests designed to verify the "Inbound Adapters" of the application. These validate HTTP routes, request/response payload serialization (JSON), status codes, authentication middlewares, and error handling. They call the actual controllers while mocking the module's Facade.

### Why?
- **Ensuring the Contract:** Guarantees that the API matches external expectations (e.g., OpenAPI/Swagger documentation) and doesn't unexpectedly break frontend consumers.
- **Isolating the Transport Layer:** By mocking the business logic (Facade), you only test if the routing and parsing behavior is correct. If an HTTP 400 Bad Request test fails, you know the issue is in the request validation, not the domain logic.
- **Graceful Failure:** Verifies that internal domain throwables or exceptions are correctly caught by error handlers and translated into appropriate, safe HTTP responses (like 404 or 500) without leaking stack traces.

---

## Infrastructure Tests (Database / External APIs)
Focused "Outbound Adapter" tests that communicate with real, external resources. For a database, this involves spinning up a real, ephemeral database instance (e.g., via Docker / Testcontainers), running the actual SQL/ORM queries, and verifying data persistence. For external APIs (like LLMs), it involves testing the HTTP clients either against a dedicated sandbox environment or using network-level interceptors.

### Why?
- **Mocks Can Lie:** A mocked database repository will always return what you tell it to. It won't tell you if your actual SQL generated by an ORM is invalid, missing a `JOIN`, or failing a unique constraint.
- **Migration Confidence:** Validates that schema migrations apply successfully and that the application code is perfectly synchronized with the current state of the database design.
- **Handling Real-World Flakiness:** External APIs fail, rate-limit, and timeout. Testing these Outbound Adapters ensures your code gracefully handles retries, malformed external responses, and network failures.
